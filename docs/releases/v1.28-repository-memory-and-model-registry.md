# v1.28 - Repository Memory (Phase 2) and Model Registry (Phase 3)

## Summary

Two phases of the HIVE production upgrade programme, delivered together:

- **Repository Memory** persists structured, queryable knowledge about a
  repository (Project DNA, architecture summary, coding standards, build and
  deployment profiles, environment schema, known issues, learned patterns,
  previous patches, optimisation history, QA history, Repository Council
  history) so it survives after the Phase 1 temporary working copy is
  cleaned up, and is queryable via Cloudflare AI Search without reloading the
  repository.
- **Model Registry** is a dynamic, ranked model list per category (coding,
  reasoning, planning, vision, research, fast, cheap, creative, long
  context). The highest-ranked model in the "coding" category automatically
  becomes the default coding model.

## Repository Memory (Phase 2)

- Built entirely on the existing `hive_ecosystem_metadata` D1 table
  (lane/source_type/source_id/metadata_json) â€” **no new schema or
  migration**. Each memory field is one row: `lane="repository_memory"`,
  `source_type=<field name>`, `source_id=<repository_id>`.
- Scalar fields (overwrite semantics): `project_dna`, `architecture_summary`,
  `coding_standards`, `build_profile`, `deployment_profile`,
  `environment_schema`.
- History fields (append-only, capped at `max_entries`, default 200):
  `known_issues`, `learned_patterns`, `previous_patches`,
  `optimisation_history`, `qa_history`, `repository_council_history`.
- `get_repository_memory()` returns every field for a repository in a single
  D1 read, without touching the repository's local working copy.
- A new `AiSearchClient` adapter (`backend/app/storage/ai_search.py`) wraps
  Cloudflare AI Search (instance `hive-repositories` by default) for semantic
  querying. **The exact AutoRAG/AI Search REST path should be reconfirmed
  against current Cloudflare docs before production use** â€” it was not
  possible to verify this live in the sandbox used to write it (no network
  access), so the endpoint is implemented defensively but unverified.

### New endpoints

| Endpoint | Purpose |
|---|---|
| `GET /v1/repositories/{id}/memory` | All Repository Memory fields for a repository |
| `GET /v1/repositories/{id}/memory/{field}` | One field |
| `PUT /v1/repositories/{id}/memory/{field}` | Set a scalar field |
| `POST /v1/repositories/{id}/memory/{field}/append` | Append a history entry |
| `GET /v1/repositories/{id}/memory-search` | D1 LIKE-search scoped to one repository |
| `GET /v1/repository-memory/ai-search` | Cloudflare AI Search semantic query |
| `GET /v1/repository-memory/ai-search/diagnostics` | AI Search config/health |

## Model Registry (Phase 3)

- `backend/app/services/model_registry.py`: in-process registry,
  `register_model(category, model_id, score=...)`, re-ranked highest-score
  first on every write; `get_default_model(category)` returns the top model
  or `None` if the category is empty.
- **Backwards compatible by design**: `ModelRouter.select_model` now checks
  `model_registry.get_default_model("coding")` first for `TaskType.CODE`, and
  only falls back to the existing `settings.code_model` when the registry has
  no entries. No other task type or existing config field changed.
- Optional `MODEL_REGISTRY_SEED_JSON` env var seeds the registry at startup
  (`main.py` lifespan calls `model_registry.seed_from_json(...)`); malformed
  or unknown-category entries are skipped rather than crashing boot.

### New endpoints

| Endpoint | Purpose |
|---|---|
| `GET /v1/model-registry/categories` | List the 9 fixed categories |
| `GET /v1/model-registry` | Every category's ranked models |
| `GET /v1/model-registry/{category}` | Ranked models + default for one category |
| `GET /v1/model-registry/{category}/default` | Just the default model_id |
| `POST /v1/model-registry/{category}` | Register/re-score a model |
| `DELETE /v1/model-registry/{category}/{model_id}` | Remove a model |

## Deferred to later phases

- Persisting the Model Registry itself in D1 (currently in-process only â€”
  resets on restart unless seeded via `MODEL_REGISTRY_SEED_JSON`); durable
  persistence plus promotion/rollback belongs to Phase 5 (AI Council) and
  Phase 11 (Optimisation Engine).
- Embedding Repository Memory documents into Cloudflare AI Search
  automatically on write (currently AI Search is a separate query surface
  the caller populates directly); wiring that indexing pipeline is part of
  Phase 2's remaining scope alongside Phase 5/6.
- Extending the registry-preferred-default pattern from "coding" to the
  other 8 categories in `ModelRouter` once there's a clear category-to-role
  mapping for summary/audit/premium tasks.
