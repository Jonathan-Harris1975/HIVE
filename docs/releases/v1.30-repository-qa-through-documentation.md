# v1.30 - Repository QA through Documentation (Phases 7-14)

## Summary

The remaining eight phases of the HIVE production upgrade programme,
completing all 14 phases.

## Phase 7 - Repository QA (`services/repository_qa.py`)

**Read this before trusting a "clean" QA report**: Repository QA is
deliberately *static-only*. It never installs dependencies or executes an
uploaded repository's build/test commands — uploaded ZIPs are untrusted
input, and running them inside HIVE's own backend process would be an
arbitrary-code-execution risk. Ten checks, each documented in-code with
exactly what it does and doesn't do:

| Check | What it really does |
|---|---|
| `build_verification` | `py_compile`s every `.py` file (real); JS/TS build tooling is presence-detected only, not run |
| `lint` | Heuristic (trailing whitespace, long lines, mixed tabs/spaces) — not ruff/eslint |
| `type_checking` | Heuristic annotation-coverage ratio — not mypy/pyright |
| `dependency_validation` | Reuses the Phase 1 manifest's dependency scan |
| `import_validation` | `ast`-parses Python imports, flags unresolved depth-1 relative imports |
| `dead_code_detection` | Reuses the existing `repo_hygiene_report` (duplicate/orphan/generated-artifact scan) |
| `security_scanning` | Regex heuristics for obviously-committed secrets — not a real secret scanner |
| `regression_testing` | Counts test files/dirs; **never executes tests** |
| `patch_verification` | Confirms the working copy fingerprint hasn't drifted since Phase 1 registration |
| `architecture_validation` | Flags conflicting Python dependency-manifest markers |

New endpoint: `POST /v1/repositories/{id}/qa` (also persists into Repository
Memory's `qa_history` via the Phase 2 `append_history_entry`).

## Phase 8 - Repository Council (`services/repository_council.py`)

Nine weighted dimensions (architecture, documentation, dependencies,
technical debt, security, performance, maintainability, ai_generated_code,
repository_health), most derived directly from the Phase 7 QA report.
**Two dimensions are explicit low-confidence placeholders**:
`performance` (no runtime profiling data exists for an uploaded repo — it's
a rough dead-code/orphan-count proxy) and `ai_generated_code` (a marker-grep
heuristic; absence of a marker does not mean code wasn't AI-generated).
Weights are configurable; history persists via Repository Memory's
`repository_council_history`.

New endpoints: `POST /v1/repositories/{id}/council`,
`GET /v1/repositories/{id}/council/history`.

## Phase 9 - Bucket Manager (`services/bucket_manager.py`)

The explicit accessible/hidden bucket lists from the programme spec, plus
`assert_accessible()` / `filter_hidden()` guards. New endpoint:
`GET /v1/buckets` (only ever returns the accessible list).

## Phase 10 - Connector Framework (`services/connectors/`)

Four initial connectors, each wrapping an existing integration rather than
reimplementing it: `openrouter_connector` (wraps the Phase 4 provider),
`r2_connector` (wraps `R2Storage`), `ai_search_connector` (wraps the Phase 2
adapter), and `github_connector` (new — minimal REST client for repo
metadata + rate limit, via `GITHUB_TOKEN`/`GITHUB_REPOSITORY`).

New endpoint: `GET /v1/connectors`.

## Phase 11 - Optimisation Engine (`services/optimisation_engine.py`)

Every optimisation decision is recorded with `previous_state`, `new_state`,
and a confidence score, and can be rolled back. **Scope note**: this is the
ledger of what to revert to, not the actuator — actually re-applying a
reverted state to whatever real system it touched is the caller's job.
Also tracks named experiments and reports success/rollback rates.

New endpoints: `POST /v1/optimisation/decisions`,
`GET /v1/optimisation/decisions`,
`POST /v1/optimisation/decisions/{id}/rollback`,
`POST /v1/optimisation/experiments`, `GET /v1/optimisation/experiments`,
`GET /v1/optimisation/stats`.

## Phase 12 - Repository Learning (`services/repository_learning.py`)

Builds entirely on Phase 2's Repository Memory fields — no new storage.
Records patch outcomes and coding patterns (including repository-scoped
model preferences, recorded as a coding pattern rather than duplicating
Phase 3's global Model Registry), then `update_project_dna()` rolls that
history plus the latest QA/Council scores up into a refreshed `project_dna`
summary field.

New endpoints: `POST /v1/repositories/{id}/learning/patch-outcome`,
`POST /v1/repositories/{id}/learning/coding-pattern`,
`POST /v1/repositories/{id}/learning/preferred-model`,
`POST /v1/repositories/{id}/learning/refresh-project-dna`.

## Phase 13 - Environment (`services/env_audit.py`)

Cross-checks every `Settings` field's declared environment variable
name(s) against `.env.example`. **This audit caught a real gap while
building Phase 10**: `GITHUB_TOKEN`/`GITHUB_REPOSITORY` had been added to
`Settings` but not yet to `.env.example` — fixed before this release. A
manual full-config cross-check (done by hand in this sandbox, since
`pydantic` isn't installable here — see Validation section) also surfaced
27 pre-existing fields from *before* this programme that were never in
`.env.example` (e.g. `POSTGRES_*`, `D1_*`, `MAST_HEALTH_URL`,
`RAMS_API_KEY`). None of those are new in this release; they're flagged
here for your awareness rather than silently fixed, since touching
pre-existing unrelated infrastructure config is outside this programme's
scope ("HIVE backend repository only" / "never redesign working systems").

New endpoint: `GET /v1/environment/audit`.

## Validation performed in this sandbox

- `python -m py_compile` on every new/modified file (clean).
- Full hand-traced test suites per phase, using the same `Fake*` D1-store
  double convention as earlier phases, plus `httpx.MockTransport` for the
  GitHub connector (matching the existing test pattern).
- Repository QA tests register a *real* repository via Phase 1's
  `register_repository` and run genuine `py_compile`/regex/`ast` checks
  against real temp files (not mocked) — these are true unit tests of real
  logic, not just interface checks.
- Manually cross-checked every `AliasChoices(...)` env var name in
  `config.py` against `.env.example` with a standalone script (since
  `pydantic` can't be installed here to run `env_audit.py` itself against
  the live `Settings` model) — this is what caught the `GITHUB_TOKEN` gap
  and the 27 pre-existing gaps noted above.

**Not run here (no network in this sandbox):** `pytest`, `ruff`, and a live
app boot, and therefore `env_audit.py`'s own `audit_environment()` was
never executed for real against the live `Settings.model_fields` — the
Phase 13 cross-check above was done by an equivalent standalone regex
script instead, which is a reasonable but not identical substitute. Run
before merging:

```bash
python -m pip install -r requirements.txt -r requirements-dev.txt
PYTHONPATH=backend python -m pytest backend/tests -q
python -m ruff check backend/app backend/tests scripts
curl -s http://localhost:8000/v1/environment/audit   # after the app is running
```

## What's still explicitly out of scope

- A real benchmark data source for AI Council (Phase 5/6 caveat, unchanged).
- Actually executing an uploaded repository's build/lint/type/test commands
  — deliberately excluded from Repository QA for security reasons. If HIVE
  ever wants real dynamic CI, it needs an isolated sandboxed runner kept
  separate from this static pipeline, not an extension of it.
- Fixing the 27 pre-existing `.env.example` gaps found outside this
  programme's file scope.
- Persisting the Model Registry and Optimisation Engine's live in-process
  state to D1 (both already write history to D1, but the ranked/decision
  state itself resets on restart unless seeded).
