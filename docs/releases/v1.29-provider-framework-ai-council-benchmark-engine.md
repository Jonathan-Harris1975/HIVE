# v1.29 - Provider Framework (Phase 4), AI Council (Phase 5), Benchmark Engine (Phase 6)

## Summary

Three phases delivered together because Phase 5 depends on Phases 4 and 6:

- **Provider Framework**: a uniform adapter shape (models, pricing, context,
  tool/structured-output support, health, latency) so no provider-specific
  code exists outside `app/services/providers/*`. OpenRouter is wrapped
  as-is (no duplicated HTTP/caching logic); any future OpenRouter-compatible
  provider is added purely through configuration.
- **Benchmark Engine**: configurable weighted scoring across 10 metric axes
  (coding/reasoning benchmarks, cost, latency, reliability, long-context,
  JSON reliability, structured output, community maturity, internal
  historical performance). Weights are fully overridable via
  `BENCHMARK_WEIGHTS_JSON`.
- **AI Council**: one `run_council()` call that discovers every configured
  provider, refreshes catalogues, diffs against the last snapshot (new/
  retired models), scores coding-capable models, auto-promotes those above
  `AI_COUNCIL_PROMOTION_THRESHOLD` into the Phase 3 Model Registry, and
  notifies downstream services through the existing ops-event inbox.

## Important honesty note on AI Council scoring

There is no live coding/reasoning benchmark data source (no HumanEval,
SWE-bench, etc. integration) wired into this sandbox or this codebase yet.
`_metrics_for_model` in `ai_council.py` only derives what a provider's
`/models` response actually exposes: pricing → cost score, context length →
long-context score, declared `supported_parameters` → structured-output
score. The other six metric axes (coding_benchmark, reasoning_benchmark,
reliability, latency, json_reliability, community_maturity,
internal_historical_performance) are left unset and the Benchmark Engine
treats missing axes as neutral (0.5) — meaning current promotion decisions
are a placeholder based on price/context/declared-capabilities, not real
benchmark performance. Wire a real benchmark data source into
`_metrics_for_model` before trusting promotions in production. This is
flagged in-code as well, not just here.

Similarly, "coding candidate" detection (`_is_coding_candidate`) is a
keyword + tool-support heuristic (`AI_COUNCIL_CODING_KEYWORDS`), not a
trained classifier — expect it to both under- and over-select models until
Phase 7/8 (Repository QA / Council) or a real benchmark integration
provides better signal.

## Provider Framework (Phase 4)

- `providers/base.py`: `ProviderAdapter` protocol, `ProviderModelInfo`,
  `ProviderHealth`, and `parse_provider_model()` — the one place that parses
  an OpenRouter-shaped model payload into capability flags, shared by every
  adapter so parsing logic isn't duplicated per provider.
- `providers/openrouter_provider.py`:
  - `OpenRouterProvider` wraps the existing `OpenRouterClient` (no
    reimplemented caching/retry).
  - `OpenRouterCompatibleProvider` is a generic adapter for any future
    provider with an OpenRouter-shaped `/models` endpoint; accepts an
    optional `transport` for testing (mirrors the `httpx.MockTransport`
    pattern already used elsewhere in the test suite).
- `providers/registry.py`: `discover_providers(settings)` — OpenRouter is
  included automatically when `OPENROUTER_API_KEY` is set; additional
  providers come from `PROVIDER_FRAMEWORK_EXTRA_PROVIDERS_JSON`.

### New endpoints

| Endpoint | Purpose |
|---|---|
| `GET /v1/providers` | List discovered providers |
| `GET /v1/providers/health` | Latency + model count per provider |
| `GET /v1/providers/{name}/models` | Normalised model list for one provider |

## Benchmark Engine (Phase 6)

- `benchmark_engine.py`: `score_model()`, `rank_models()`,
  `load_weights()` / `normalise_weights()`. Missing metrics default to a
  neutral 0.5 (not 0.0) so partial benchmark coverage isn't unfairly
  punished; `confidence` reports how many of the 10 axes were actually
  supplied.

### New endpoints

| Endpoint | Purpose |
|---|---|
| `GET /v1/benchmark/metrics` | The 10 supported metric keys |
| `POST /v1/benchmark/rank` | Rank arbitrary candidate metrics, optional custom weights |

## AI Council (Phase 5)

- `ai_council.py`: `run_council(settings)` returns a `CouncilRunReport`
  (providers discovered, models seen, new/retired model IDs, promotions,
  weights used). Persisted via the existing `hive_ecosystem_metadata` D1
  table (lane=`ai_council`) — no new schema. One provider raising an
  exception during a run does not sink the whole council run (caught and
  skipped).
- Promotions call `model_registry.register_model("coding", ...)` (Phase 3)
  and push a `model_promotion` event through `ops_events.ingest_ops_event`
  (Phase already-existing inbox used by GitHub/provider watchers), so MAST
  or AIMS can react without polling HIVE directly.

### New endpoints

| Endpoint | Purpose |
|---|---|
| `POST /v1/ai-council/run` | Run discovery + benchmark + promotion now |
| `GET /v1/ai-council/history` | Past run reports (new/retired models, promotions) |

## Validation performed in this sandbox

- `python -m py_compile` on every new/modified file (clean).
- `OpenRouterCompatibleProvider` tests use `httpx.MockTransport` (the same
  pattern as `test_repo_health_dashboard.py`), so no real network call is
  made or required.
- AI Council tests use fake in-memory `ProviderAdapter` and D1 store doubles
  (monkeypatched `discover_providers` / `D1MetadataStore` inside the
  `ai_council` module), covering: promotion above threshold, no promotion
  for non-tool-capable/low-scoring models, new/retired model diffing across
  two runs, ops-event notification, run-history accumulation, and a
  provider that raises not sinking the whole run.
- Confirmed no route-shadowing between literal (`/providers/health`) and
  parameterised (`/providers/{provider_name}/models`) paths — different
  segment counts, so no ambiguity regardless of registration order.

**Not run here (no network in this sandbox):** `pytest`, `ruff`, and a live

## Deferred to later phases

- A real benchmark data source (see honesty note above) — currently the
  Benchmark Engine is generic and correct, but AI Council only feeds it
  price/context/capability-derived metrics.
- Scheduling `run_council()` monthly (currently on-demand via
  `POST /v1/ai-council/run` — wire into MAST or a cron-style job).
- Persisting the Model Registry itself in D1 (still in-process only,
  per the Phase 3 release note) so AI Council promotions survive restarts
  without re-running the council.
- Extending AI Council beyond the "coding" category once there's a
  clear signal for classifying models into the other 8 categories.
